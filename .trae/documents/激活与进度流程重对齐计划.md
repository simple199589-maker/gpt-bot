# 激活与进度流程重对齐计划

## 目标

基于当前已还原的代码状态，完成以下三类适配：

1. 菜单文案重新对齐到当前 bot 实际菜单。
2. 激活请求发送后的新返回文案分类重构。
3. `💰 查余额/进度` 返回的新进度页文案兼容。

本次只做最小必要改动，不改接口路径，不做无关重构。

---

## 现状判断

当前代码仍使用旧菜单与旧文案：

- `activate_plus` 对应 `⚡️ 激活plus母号`
- `activate_team` 对应 `👥 激活team母号`
- `balance` 对应 `💰 查余额`

当前激活处理中与终态识别也还是旧逻辑：

- “当前状态”“次查询”“请稍候”等被直接归到 progress
- 未覆盖“请求已加入队列”
- 未覆盖“你已经有一个任务在队列中”
- 未覆盖新的 `查余额/进度` 结构化状态页

因此如果直接执行当前代码，至少会出现：

1. 点不到正确菜单。
2. 激活已接单消息无法被正确视为“已接受请求”。
3. “已存在排队任务”可能被误判。
4. `查余额/进度` 无法稳定命中新文案，也无法提取余额点数。

---

## 方案原则

### 方案选择

采用 **最小改动方案**：

- 保持现有 API 结构不变。
- 保持 `WorkflowResult` 结构不变。
- 先通过关键词和分类顺序适配新文案。
- 余额/进度页先保证“能命中 + 能提取余额 + 保留原始文本”。
- 暂不新增复杂结构化字段，除非后续再明确需要。

### 不采用的方案

不在本轮直接引入新的响应模型字段，例如：

- `queue_position`
- `queue_wait_minutes`
- `daily_success_used`
- `daily_success_limit`
- `extra_charge_count`

原因：这会牵涉 `schemas.py`、`api_server.py`、响应模型兼容和更多测试，不符合当前“先恢复可用”的目标。

## 实施步骤

需要同步修改以下配置来源中的按钮映射：

- `app_config.py` 的 `DEFAULT_CONFIG`
- `config.example.json`
- 如存在实际运行配置 `config.json`，也同步修改

目标菜单映射调整为：

- `activate_plus` → `⚡️ Plus直充`
- `activate_team` → `🚀 Pro 20x直充`
- `balance` → `💰 查余额/进度`
- `redeem` → `🎟 兑换卡密`

同时检查 README 中涉及旧菜单文案的说明，按最小范围同步。

---

### 第二步：重构激活返回文案识别

#### 2.1 新增“已接受请求”的处理中识别

当发送请求后，若返回：

- `⏳ 你的请求已加入队列。`
- `排队位置：第 xx 位`
- `预计等待：约 xx 分钟`
- `请不要重复提交`
- `随时点击「💰 查余额/进度」可查看实时排队进度`

则应视为：

- 请求已被接受
- 当前状态属于 `processing`
- 应立即通过现有 `progress_callback` 返回给上层
- 不应被当作失败

实现方式：

- 在 `activation_progress` 关键词中新增上述命中特征。
- 保证该类文案能被 `_classify_activation_message()` 归为 `progress`。

#### 2.2 新增“已重复提交/已有排队任务”的处理中识别

当返回：

- `你已经有一个任务在队列中。`
- `当前状态：排队中`
- `排队位置：第 xx 位`
- `请等待完成后再提交新的请求。`

这类消息语义上表示：

- 系统未接受新的重复任务
- 但已有任务正在队列中
- 对当前调用方来说，本质上仍然属于“继续等待，不应再次提交”

本轮建议仍归类为：

- `processing`

原因：

- 这比归为失败更贴合“已有任务继续等待”的业务语义。
- 与你描述的“只用等即可”一致。
- 不需要新增新的 API 状态枚举。

实现方式：

- 为 `activation_progress` 增加：
  - `你的请求已加入队列`
  - `你已经有一个任务在队列中`
  - `排队中`
  - `排队位置`
  - `预计等待`
  - `请不要重复提交`
  - `请等待完成后再提交新的请求`
  - `实时排队进度`

#### 2.3 保留原有成功/失败/取消终态判断

以下逻辑继续保留：

- 成功：`升级成功` / `激活成功` 等
- 失败：`Token 无效` / `额度已退回` / `激活失败` 等
- 取消：`已取消`

但需要检查分类顺序，避免：

- 某些带 `当前状态` 的排队消息被错误提前归类
- 某些失败消息因带进度词而被误判为 progress

建议顺序：

1. 先判终态关键词（取消 / 成功 / 失败）
2. 再判显式失败规则
3. 再判显式成功规则
4. 最后判 progress

这样能避免误判。

### 第三步：适配 `查余额/进度` 新文案

#### 3.1 扩展 balance 关键词

新的 `查余额/进度` 页面可能不只包含单行余额，而是完整状态页。

需要把以下关键词纳入 `result_keywords.balance`：

- `额度余额：`
- `额度余额:`
- `当前队列`
- `你的队列任务`
- `今日已成功：`
- `额外直充次数：`
- `当前状态：排队中`
- `前方还有`
- `预计等待：`

这样查询接口能稳定命中新页面。

#### 3.2 扩展余额提取逻辑

当前 `_extract_balance()` 只支持：

- `余额：12`

需要扩展为同时支持：

- `额度余额：6 点`
- `额度余额:6 点`
- 保留兼容旧格式 `余额：12`

实现方式：

- 优先匹配 `额度余额[：:]\s*(\d+)\s*点`
- 再兼容 `余额[：:]\s*(\d+)`

返回仍然写入 `WorkflowResult.balance`。

#### 3.3 保留完整原始文本

虽然这次不新增结构化字段，但仍保持：

- `message` = 完整原始文案
- `raw_message` = 完整原始文案

这样上层如果暂时需要拿排队位置、预计等待等信息，可以先从原文中取。

## 预期结果

完成后应达到以下效果：

### 菜单层面

能够正确点击：

- `⚡️ Plus直充`
- `🚀 Pro 20x直充`
- `💰 查余额/进度`
- `🎟 兑换卡密`

### 激活层面

发送 token 后：

- 若机器人返回“你的请求已加入队列”，系统立即返回 `processing`
- 若机器人返回“你已经有一个任务在队列中”，系统也返回 `processing`
- 若机器人返回成功文案，系统返回成功
- 若机器人返回失败文案，系统返回失败

### 查余额/进度层面

当返回完整状态页时：

- 接口能命中该消息
- 能提取 `额度余额`
- `message/rawMessage` 保留完整页面内容

## 风险与注意项

1. `当前状态` 既可能出现在排队页，也可能出现在别的消息中，因此分类顺序必须谨慎。
2. `你已经有一个任务在队列中` 从业务上是“重复提交被拒绝”，但从对接语义上应视为“继续等待中的处理中”，本轮按 `processing` 处理。
3. 本轮不新增结构化进度字段，因此调用方若要读取排队位置仍需先解析原文。
4. 如果后续你确认需要精确对接队列位置、前方人数、预计等待、每日次数等，再单独做第二轮结构化扩展。
